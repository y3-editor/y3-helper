## Context

Y3 Helper 扩展当前使用 `scriptUri`（脚本目录，通常是 `script/` 文件夹）作为 VSCode 工作区根目录。这个设计源于早期对 Lua 脚本开发的聚焦，但随着项目复杂度增加，用户需要访问项目级别的资源，如 global script、配置文件、资源目录等。

**当前状态**：
- `selectAnotherMap` 命令切换地图后，会将工作目录设为 `scriptUri`
- 用户无法直接在 VSCode 文件树中看到脚本目录之外的内容
- `y3.env` 已提供 `projectUri` 属性，但未被充分利用

**约束**：
- 必须保持向后兼容，不影响现有用户的基本工作流
- 需要考虑 VSCode 窗口重启对用户体验的影响
- 国际化支持（中英文）

## Goals / Non-Goals

**Goals:**
- 将工作目录切换逻辑从脚本级别提升到项目级别
- 在地图管理页面提供「打开地图根目录」的快捷入口
- 保持 UI 一致性，复用现有的 `ViewInVSCode` 组件

**Non-Goals:**
- 不修改 `env.ts` 中的环境变量获取逻辑
- 不添加多工作区（Multi-root Workspace）支持
- 不修改其他使用 `scriptUri` 的功能模块

## Decisions

### Decision 1: 工作目录切换目标

**选择**: 使用 `projectUri` 替代 `scriptUri` 作为工作目录

**理由**:
- `projectUri` 包含完整的项目结构，用户可以访问 global script、资源文件等
- `scriptUri` 是 `projectUri` 的子目录，切换后用户仍可访问脚本
- 不需要修改环境变量逻辑，`y3.env.projectUri` 已存在

**替代方案**:
- 多工作区支持：复杂度高，需要处理多根目录的文件解析逻辑
- 配置项让用户选择：增加配置复杂度，大多数用户期望项目级别访问

### Decision 2: 菜单项位置与实现

**选择**: 在地图管理页面的分割线下方添加「打开地图根目录」菜单项

**理由**:
- 复用现有的 `ViewInVSCode` 组件，保持代码一致性
- 放在「打开脚本目录」之前，强调项目级别优先
- 分割线区分功能区块，UI 结构清晰

**替代方案**:
- 添加到命令面板：不够直观，用户需要记住命令名
- 添加到工具栏：位置有限，地图管理页面更合适

### Decision 3: 国际化处理

**选择**: 在 `l10n/bundle.l10n.json` 中添加中英文翻译

**理由**:
- 遵循项目现有的国际化模式
- 使用 `l10n.t()` 函数保持一致性

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 用户习惯改变：工作目录变化可能导致用户困惑 | 提供明确的菜单项说明和 tooltip 提示 |
| VSCode 窗口重启：打开新目录会导致窗口重新加载 | 在 tooltip 中明确提示「会重启 VSCode 窗口」 |
| `projectUri` 为空：某些边界情况下可能未初始化 | 添加 `if (y3.env.projectUri)` 条件检查 |
