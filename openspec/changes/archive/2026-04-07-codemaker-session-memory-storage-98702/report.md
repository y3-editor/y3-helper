**核心痛点**: CodeMaker 本地 api-server 使用单文件 `chat_histories.json` 存储所有会话数据（含完整消息内容），随使用时间增长文件不断膨胀，导致启动慢、保存慢、内存浪费。前端已实现分页/排序/过滤等查询参数，但本地 api-server 全部忽略。

**落地成效**: 将存储方案从"单文件全量内存"重构为"索引文件 + 单会话文件"结构，对齐 CodeMaker 云端服务设计。1 次提交，511 行代码变更，仅修改 1 个文件（history.mjs），前端零修改。支持旧数据自动迁移，测试验证通过。

**关键方法**: 从 CodeMaker 开源前端源码反向追踪 API 调用链，发现前端已实现 `_exclude`/`_page`/`_num`/`_sort_by` 等参数但被本地版忽略。参照云端 API 设计理念，在本地 api-server 实现等效的索引+按需加载机制。预埋 `_schema_version` 字段为后续"一键上传对话"需求铺路。

**[解决分享]**: 本次重构验证了"前端代码即文档"的思路——通过阅读前端已实现但未使用的查询参数，反推出后端应有的能力。改动范围极小（1个文件），但架构收益明显：启动快、保存快、内存省、为后续功能铺路。
